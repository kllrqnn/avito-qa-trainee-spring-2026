# Тест-кейсы для проверки API объявлений (Avito QA Trainee)

**Host:** `https://qa-internship.avito.com`

## 1. Создание объявления (POST /api/1/item)

| ID | Название | Шаги | Ожидаемый результат |
|:---|:---|:---|:---|
| CP-1 | Создание с полным набором данных | POST `/api/1/item` с валидными `name`, `price`, `sellerId`, `statistics` | 200 OK. JSON содержит `id` (string), `createdAt` (string) и эхо-данные |
| CP-2 | Создание с минимальным набором данных | POST `/api/1/item` только с `name`, `price`, `sellerId` | 200 OK. Объявление создано успешно |
| CN-1 | Создание: Невалидный тип `price` | POST `/api/1/item` где `price` — строка | 400 Bad Request |
| CN-2 | Создание: Отрицательная цена | POST `/api/1/item` с `price: -1` | 400 Bad Request (или логическая ошибка в BUGS.md) |
| CN-3 | Создание: Пропуск обязательного поля | POST `/api/1/item` без `sellerId` | 400 Bad Request |

## 2. Получение объявления по ID (GET /api/1/item/:id)

| ID | Название | Шаги | Ожидаемый результат |
|:---|:---|:---|:---|
| RP-1 | Получение существующего объявления | GET `/api/1/item/{id}` | 200 OK. Тело — массив `[{...}]`. Данные внутри совпадают с отправленными при создании |
| RN-1 | Получение по несуществующему ID | GET `/api/1/item/non-existent-uuid` | 404 Not Found |
| RN-2 | Получение по ID в неверном формате | GET `/api/1/item/12345` (не UUID) | 400 Bad Request |

## 3. Получение всех объявлений продавца (GET /api/1/:sellerID/item)

| ID | Название | Шаги | Ожидаемый результат |
|:---|:---|:---|:---|
| SP-1 | Получение списка объявлений продавца | GET `/api/1/{sellerId}/item` | 200 OK. Массив объектов. Все объекты имеют указанный `sellerId` |
| SP-2 | Запрос для продавца без объявлений | GET `/api/1/999999999/item` | 200 OK. Пустой массив `[]` |

## 4. Статистика (GET /api/1/statistic/:id)

| ID | Название | Шаги | Ожидаемый результат |
|:---|:---|:---|:---|
| ST-1 | Получение статистики | GET `/api/1/statistic/{id}` | 200 OK. Объявление содержит поля `likes`, `viewCount`, `contacts` |

## 5. Расширенные проверки (Корнер-кейсы)

| ID | Название | Шаги | Ожидаемый результат |
|:---|:---|:---|:---|
| E2E-1 | Полный цикл: Создание -> Проверка | 1. POST создание. 2. GET по ID. 3. GET по sellerId | На всех этапах данные консистентны. Объявление доступно во всех ручках |
| IDM-1 | Идемпотентность GET | Выполнить GET `/api/1/item/{id}` 3 раза подряд | Результаты и статус-коды идентичны. Состояние системы не изменилось |

## 6. Нефункциональные проверки
| ID | Название | Описание | Ожидаемый результат |
|:---|:---|:---|:---|
| PERF-1 | Время ответа (Performance) | Замерить время выполнения GET запроса | Ответ приходит быстрее, чем за 500мс (стандарт для микросервисов) |
| SEC-1 | Заголовки ответа (Security) | Проверить наличие заголовка Content-Type | Значение: `application/json` |

## 7. Другие важные аспекты
| ID | Название | Описание | Ожидаемый результат |
|:---|:---|:---|:---|
| DATA-1 | Формат даты (createdAt) | Проверить, что дата создания соответствует ISO 8601 | Валидный формат даты и времени |
